# 基於 FPGA 之智慧型感應紅綠燈控制系統－專案計畫書

## 一、專案基本資料

- 專案名稱：基於 FPGA 之智慧型感應紅綠燈控制系統
- 開發平台：Terasic DE2-115
- FPGA：Intel/Altera Cyclone IV E EP4CE115F29C7
- 開發工具：Quartus II 13.1 Web Edition
- 硬體描述語言：Verilog HDL（Verilog-2001，原始碼使用 `.v`）
- 預計使用介面：`CLOCK_50`、`KEY[3:0]`、`SW[17:0]`、`LEDR[17:0]`、`LEDG[8:0]`、`HEX7`～`HEX4`、LCD

## 二、專案摘要

本專題將使用 DE2-115 FPGA 實驗板，實作一套簡化的十字路口智慧型紅綠燈控制系統。系統以有限狀態機控制東西向與南北向燈號，除了基本的紅、黃、綠燈循環外，也加入行人過街請求、車流感應、夜間閃燈與故障安全模式。LCD 用來顯示目前燈號、運作模式及剩餘秒數，並提供可由按鍵操作的參數設定選單。

本專題的重點是運用 FPGA 的計數器、除頻器、有限狀態機、輸入同步及 LCD 控制等數位邏輯技術，完成一套具備基本安全邏輯與簡易智慧調節能力的交通號誌系統。

本專案所有設計模組皆使用 Verilog HDL，不使用 SystemVerilog 的 `logic`、`always_ff`、`always_comb`、`typedef enum` 等語法，以確保與 Quartus II 13.1 的工程設定相容。

## 三、研究動機

傳統定時紅綠燈依固定時間循環，無法根據道路是否有車或是否有行人等待而調整，可能造成不必要的等候。本專題以開關模擬車流感測器、按鍵模擬行人請求，讓交通號誌能依輸入狀況調整綠燈時間，呈現智慧交通控制的基本概念。

FPGA 具有平行處理、時序精確及硬體邏輯可重複配置等特性，適合用來實作即時交通號誌控制器。透過本專題可練習從需求分析、狀態設計、Verilog 撰寫，到實驗板驗證的完整數位系統開發流程。

## 四、專案目標

1. 完成東西向與南北向交通燈的安全循環控制。
2. 以按鍵模擬行人提出過街請求，並在安全時機安排通行。
3. 以開關模擬道路車流感測器，依需求延長綠燈時間。
4. 提供夜間閃燈模式與故障安全模式。
5. 使用 LCD 顯示目前模式、燈號狀態與倒數時間。
6. 使用七段顯示器分別顯示東西向及南北向的紅／綠燈剩餘時間。
7. 使用 `SW[17]` 進入 LCD 設定選單，並透過 `KEY[3:0]` 調整運行參數。
8. 完成 Quartus 編譯、腳位配置及 DE2-115 實板測試。

## 六、功能說明

### 6.1 一般交通號誌循環

系統控制東西向與南北向兩組號誌，基本流程如下：

```text
東西向綠燈
  ↓
東西向黃燈
  ↓
全方向紅燈
  ↓
南北向綠燈
  ↓
南北向黃燈
  ↓
全方向紅燈
  ↓
回到東西向綠燈
```

任何時刻都不允許兩個方向同時出現綠燈。方向切換時必須先經過黃燈及全紅燈，避免直接切換所造成的危險狀態。

### 6.2 行人過街請求

- 按下行人按鍵後，系統記錄一次待處理請求。
- 系統不會立即中斷目前綠燈，而是在最低安全綠燈時間結束後切換。
- 經過黃燈及全紅燈後，進入行人通行狀態。
- 通行時間結束後清除請求，回到一般號誌循環。
- LCD 顯示 `PED REQUEST`、`PLEASE WAIT` 或 `PED GO` 等訊息。

### 6.3 車流感應模式

以兩個開關模擬東西向及南北向車輛感測器：

- 綠燈方向仍有車且另一方向無車時，可延長綠燈。
- 車流感測只延長綠燈，不縮短目前方向原有的固定綠燈時間。
- 每次綠燈皆設定最短與最長時間，避免頻繁切換或某方向長期無法通行。
- 關閉智慧感應模式時，系統恢復固定時間循環。

### 6.4 夜間閃燈模式

- 啟用後暫停一般交通號誌流程。
- 主幹道以黃燈閃爍表示注意通行。
- 支線道路以紅燈閃爍表示停車確認後通行。
- LCD 顯示 `NIGHT MODE`。
- 離開夜間模式時先回到全紅安全狀態，再重新開始一般循環。

### 6.5 故障安全模式

- 以開關模擬感測器或控制系統故障。
- 故障發生時，故障模式具有最高優先權。
- 兩個方向皆顯示閃紅燈，一般狀態機暫停。
- LCD 顯示 `SYSTEM FAULT`。
- 故障解除後先進入全紅安全狀態，再恢復一般運作。

### 6.6 系統設定模式

將 `SW[17]` 撥至 ON 時，系統提出進入設定模式的請求。為避免設定期間出現不確定燈號，控制器會先完成必要的黃燈過渡，再進入全紅安全狀態並暫停一般交通流程。LCD 隨後顯示設定選單。

按鍵在設定模式中的功能如下：

| 按鍵 | 選單瀏覽狀態 | 數值編輯狀態 |
|---|---|---|
| `KEY[0]` | 選擇目前項目／進入編輯 | 確認數值並返回選單 |
| `KEY[1]` | 游標向上移動 | 數值增加 |
| `KEY[2]` | 游標向下移動 | 數值減少 |
| `KEY[3]` | 回上一頁 | 取消本次修改並返回選單 |

設定畫面分為「設定首頁 → 項目選單 → 數值編輯」三層。在設定首頁按 `KEY[0]` 進入項目選單；項目選單按 `KEY[3]` 回設定首頁；數值編輯時按 `KEY[3]` 取消修改並回項目選單。將 `SW[17]` 撥回 OFF 才會正式離開設定模式。離開時系統先進入全紅安全狀態，再從一般交通流程重新開始。

設定首頁顯示範例：

```text
SYSTEM SETTINGS
KEY0:ENTER
```

LCD 選單採用兩行捲動顯示，例如：

```text
>GREEN TIME 10
 YELLOW TIME 3
```

進入編輯時顯示目前值與允許範圍：

```text
EDIT GREEN TIME
10s  RANGE 5-30
```

建議提供下列設定項目：

| 設定項目 | 建議預設值 | 建議範圍 | 說明 |
|---|---:|---:|---|
| 最短紅燈時間 | 14 秒 | 8～40 秒 | 確保任一方向至少維持指定紅燈時間 |
| 一般綠燈時間 | 10 秒 | 5～30 秒 | 固定循環時的基本綠燈時間 |
| 黃燈時間 | 3 秒 | 2～5 秒 | 方向切換前的警示時間 |
| 全紅安全時間 | 1 秒 | 1～3 秒 | 兩個方向皆為紅燈的安全間隔 |
| 行人通行時間 | 5 秒 | 3～15 秒 | 行人綠燈維持時間 |
| 智慧模式最短綠燈 | 5 秒 | 3～15 秒 | 有另一方向需求時，最早可切換的時間 |
| 智慧模式最長綠燈 | 15 秒 | 10～40 秒 | 持續有車時允許延長的上限 |
| 閃燈週期 | 1 秒 | 1～3 秒 | 夜間及故障模式的閃爍速度 |
| 恢復預設值 | — | — | 將所有設定還原為預設值 |

其中「最短紅燈時間」不是強制在指定秒數後立刻轉綠；若對向仍處於合法綠燈或黃燈流程，實際紅燈可以更久，但不可短於設定值。如此可保留使用者想調整紅燈時間的功能，同時避免破壞雙向號誌互鎖。

為控制期末專題規模，第一版至少完成「最短紅燈、一般綠燈、黃燈、行人通行時間、恢復預設值」五個項目；其餘項目可在核心選單穩定後加入。

設定值儲存在 FPGA 暫存器中，確認後立即更新，但離開設定模式後才套用至交通控制器。重新下載電路或重新上電後恢復預設值，本專題暫不要求寫入 EEPROM 永久保存。

### 6.7 緊急車輛模式（選做）

- 按鍵用來模擬救護車或消防車接近。
- 系統先完成黃燈及全紅過渡，再開放指定方向綠燈。
- 緊急通行時間結束後，以全紅狀態銜接回一般流程。

### 6.8 七段顯示器倒數

使用四個七段顯示器，同時呈現東西向與南北向號誌剩餘時間：

| 七段顯示器 | 顯示內容 |
|---|---|
| `HEX5` | 東西向剩餘時間十位數 |
| `HEX4` | 東西向剩餘時間個位數 |
| `HEX7` | 南北向剩餘時間十位數 |
| `HEX6` | 南北向剩餘時間個位數 |

顯示規則如下：

- 某方向為綠燈時，顯示該方向目前綠燈的剩餘秒數。
- 某方向為紅燈時，顯示該方向距離下一次轉為綠燈的預估剩餘秒數。
- 紅燈預估時間應包含對向剩餘綠燈、對向黃燈及全紅安全間隔；若有已排定的行人通行階段，也應計入。
- 智慧感應模式延長綠燈時，紅燈預估時間應同步調整；無法確定延長結束時間時顯示 `--`。
- 某方向進入黃燈時，該方向顯示黃燈剩餘秒數，LCD 同時顯示 `YELLOW`，讓使用者辨識目前燈號。
- 倒數值限制為 `00`～`99`；超過 99 秒時顯示 `99`。
- 正常倒數時，十位數為 0 可選擇顯示 `0` 或熄滅，本專題預設保留前導零，例如 8 秒顯示 `08`。
- 夜間模式、故障模式及系統設定模式無法提供一般倒數，此時兩組顯示器皆顯示 `--`。
- 未使用的 `HEX3`～`HEX0` 預設熄滅，保留作後續擴充。

DE2-115 七段顯示器為低電位點亮，解碼模組輸出時必須使用 active-low 段碼。

## 七、輸入與輸出配置

### 7.1 輸入配置

DE2-115 的 `KEY` 為低電位有效，程式中需進行反相、同步與按鍵除彈跳處理。

| 輸入 | 規劃功能 |
|---|---|
| `CLOCK_50` | 系統 50 MHz 時脈 |
| `KEY[0]` | 一般模式：系統重設；設定模式：選擇／確認 |
| `KEY[1]` | 一般模式：行人過街請求；設定模式：游標上移／增加數值 |
| `KEY[2]` | 一般模式：緊急車輛（選做）；設定模式：游標下移／減少數值 |
| `KEY[3]` | 一般模式：手動測試或 LCD 換頁（選做）；設定模式：回上一頁／取消修改 |
| `SW[0]` | 東西向有車感測 |
| `SW[1]` | 南北向有車感測 |
| `SW[2]` | 啟用智慧車流感應 |
| `SW[3]` | 啟用夜間閃燈模式 |
| `SW[4]` | 模擬系統故障 |
| `SW[17]` | 系統設定模式開關 |

### 7.2 LED 輸出配置

DE2-115 沒有獨立黃色 LED，因此同方向的紅、綠 LED 同時亮起時，定義為「黃燈」狀態；LCD 同時顯示 `YELLOW`，避免判讀不清。

| 輸出 | 規劃功能 |
|---|---|
| `LEDR[0]` | 東西向紅燈 |
| `LEDG[0]` | 東西向綠燈 |
| `LEDR[1]` | 南北向紅燈 |
| `LEDG[1]` | 南北向綠燈 |
| `LEDR[2]` | 行人停止 |
| `LEDG[2]` | 行人通行 |
| `LEDR[17:3]` | 關閉或保留作除錯顯示 |
| `LEDG[8:3]` | 關閉或保留作除錯顯示 |

燈號編碼如下：

| 邏輯燈號 | 同方向紅 LED | 同方向綠 LED |
|---|---:|---:|
| 紅燈 | 1 | 0 |
| 黃燈 | 1 | 1 |
| 綠燈 | 0 | 1 |
| 熄滅 | 0 | 0 |

| 狀態代號 | 狀態說明 |
|---|---|

| `ST_ALL_RED_1` | 切換至南北向前的全紅狀態 |
| `ST_NS_GREEN` | 南北向綠燈、東西向紅燈 |
| `ST_NS_YELLOW` | 南北向黃燈、東西向紅燈 |
| `ST_ALL_RED_2` | 切換至東西向前的全紅狀態 |
| `ST_PED_GO` | 行人通行、車輛全紅 |

| `ST_CONFIG_EXIT` | 離開設定後的安全過渡 |

### 8.2 控制優先權

系統事件優先順序規劃如下：

```text
系統重設 > 故障模式 > 系統設定 > 夜間模式 > 緊急車輛 > 行人請求 > 車流感應 > 固定循環
```

較高優先權事件發生時，仍應遵守黃燈與全紅安全過渡；只有系統重設或故障狀態可直接使輸出進入安全的全紅／閃紅狀態。若設定開關與故障訊號同時啟用，系統應停留在故障模式，待故障解除後才允許進入設定頁面。

## 九、初步時間參數

為方便實板展示，初期使用較短時間：

| 項目 | 初步設定 |
|---|---:|
| 最短紅燈時間 | 14 秒 |
| 固定綠燈時間 | 10 秒 |
| 最短綠燈時間 | 5 秒 |
| 最長綠燈時間 | 15 秒 |
| 黃燈時間 | 3 秒 |
| 全紅安全時間 | 1 秒 |
| 行人通行時間 | 5 秒 |
| 夜間／故障閃爍週期 | 1 秒 |

各項預設值、最小值與最大值以 Verilog `parameter` 宣告；使用者確認的設定值則保存在暫存器中。模擬時可縮小 `parameter` 所定義的時脈計數基準，實板測試時再恢復為一秒，無須改變選單邏輯。

設定控制器應檢查參數間的關係，例如「智慧模式最長綠燈」不得小於「智慧模式最短綠燈」。若修改後產生衝突，可自動調整另一個數值，或拒絕確認並在 LCD 顯示 `INVALID VALUE`。

## 十、系統模組規劃

```text
digital_system_final_project（Top-level）
├─ clock_divider       產生 1 秒及閃爍時基
├─ input_conditioner   按鍵同步、除彈跳與單脈衝
├─ traffic_controller  交通號誌有限狀態機
├─ traffic_timer       狀態時間與剩餘秒數管理
├─ settings_controller LCD 選單、游標、編輯值與參數檢查
├─ led_decoder         狀態轉換為 LED 輸出
├─ countdown_display   計算雙向紅／綠燈剩餘時間
├─ seven_seg_decoder   0～9、負號及熄滅的七段顯示解碼
└─ lcd_controller      LCD 初始化、字元寫入與畫面更新
```

若工期有限，可將計時器、LED 解碼及狀態機先整合在 `traffic_controller` 中，待功能穩定後再拆分模組。

| 第 6 階段 | 加入夜間與故障模式 | 特殊模式可正確進出 |
| 第 7 階段 | 完成 LCD 顯示 | 顯示模式、燈號與倒數 |
| 第 8 階段 | 加入系統設定選單與參數調整 | 可用按鍵瀏覽、修改及套用設定 |
| 第 9 階段 | 整合測試、修正與成果紀錄 | 可展示版本及報告素材 |

## 十二、測試計畫與驗收標準

### 12.1 基本功能測試

- 重設後系統必須進入明確且安全的初始狀態。
- 東西向與南北向不得同時為綠燈。
- 每次方向切換必須經過黃燈及全紅狀態。
- 狀態維持時間與 LCD 倒數須一致。
- `HEX5`、`HEX4` 須顯示東西向紅／綠燈剩餘時間。
- `HEX7`、`HEX6` 須顯示南北向紅／綠燈剩餘時間。
- 七段顯示器倒數須每秒更新，且與交通狀態及 LCD 顯示一致。
- 夜間、故障及設定模式下，`HEX7`～`HEX4` 須顯示 `--`。

### 12.2 輸入功能測試

- 長按行人按鍵只建立一筆請求，不得連續重複觸發。
- 行人請求不得讓目前綠燈立即跳成另一方向綠燈。
- 車流感應不得突破最短及最長綠燈限制。
- 夜間模式解除後須經全紅狀態再恢復循環。
- 故障模式下所有一般請求均不得造成不安全燈號。
- `SW[17]` 開啟後必須先安全轉為全紅，才可進入設定頁面。
- 設定模式中 `KEY[0:3]` 的選擇、上下移動與返回功能必須正確。
- 編輯值不得超出設定範圍，取消修改時不得改變已儲存值。
- 設定模式離開後，新值須正確套用，且交通流程須由安全狀態恢復。
- 「最長綠燈」不得小於「最短綠燈」。

### 12.3 驗收標準

1. Quartus 編譯無 Error。
2. 設計可下載至 DE2-115 並持續穩定運作。
3. LED 與 LCD 顯示符合目前狀態。
4. 所有模式切換均不產生雙向同時綠燈。
5. 行人與車流感應功能可由按鍵、開關重複展示。
6. 系統設定選單可修改至少五項參數，且所有參數皆有範圍保護。
7. 七段顯示器可同時顯示雙向剩餘時間，並正確反映智慧感應造成的時間變化。

## 十三、預期成果

- 一套可在 DE2-115 上執行的智慧型紅綠燈控制系統。
- 完整 Verilog 原始碼及 Quartus 專案設定。
- 可展示一般循環、行人請求、車流感應、夜間模式及故障模式。
- LCD 可顯示即時狀態與剩餘秒數。
- `HEX5`、`HEX4` 顯示東西向，`HEX7`、`HEX6` 顯示南北向的紅／綠燈剩餘秒數。
- 可透過 `SW[17]` 與四個按鍵操作 LCD 設定選單，調整交通號誌參數。
- 專題報告、狀態圖、測試紀錄及實板成果照片或影片。

## 十四、可能風險與應對方式

| 風險 | 應對方式 |
|---|---|
| 現有工程缺少子模組且頂層名稱不一致 | 建立獨立的 `digital_system_final_project` 頂層及本專題所需模組，不依賴原示範設計 |
| LCD 控制時序較複雜 | 先完成 LED 主功能，再單獨驗證 LCD 初始化與寫字 |
| 按鍵彈跳造成重複請求 | 使用同步器、除彈跳及單脈衝電路 |
| 實際秒數模擬耗時過長 | 將時間常數參數化，模擬時使用較小數值 |
| 特殊模式切換造成不安全燈號 | 統一經由全紅安全狀態銜接，並設定明確事件優先權 |
| 設定值互相衝突或超出合理範圍 | 為每個項目設定上下限，確認時檢查最短／最長時間關係 |
| 設定模式占用原有按鍵功能 | 以 `SW[17]` 明確切換操作情境，並在 LCD 顯示目前模式 |
| 功能過多導致無法如期完成 | 先完成必做核心，再依時間加入緊急車輛等擴充功能 |

## 十五、結論

本專題以一般紅綠燈有限狀態機為核心，加入車流、行人與安全模式，使題目兼具實用性、展示效果及數位邏輯設計內容。整體功能可分階段完成，即使不實作所有加分項目，仍能形成一套完整且容易說明的期末作品。
